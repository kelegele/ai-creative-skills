# gzh-longform 公众号长文创作

通用公众号长文 skill。读目的仓库 `AGENTS.md`/`CLAUDE.md` 取人设;不依赖第三方 skill 或付费 API;爆款规律内置 + 可选联网查证;输出 markdown。

## 两阶段工作流

研究阶段(写之前确认事实和观点)+ 写作阶段(确认后写),中间**硬闸门**。

### 阶段一:研究

**第 0 步 运行参数 + 读人设**

问用户(回答存为本次运行变量):
- **OUTPUT_DIR** — 默认 `Content/YYYYMMDD-<slug>/`(今日创建日),用户可改
- **字数** — 默认 1500,允许 ≤2000
- **是否启用联网查证** — 默认开

读目的仓库 `AGENTS.md`/`CLAUDE.md` 提取人设(详见 `references/persona-reading.md`)。关键字段缺失则停下问用户。

**第 1 步 接收选题**
- **backlog 取**:扫目的仓库 `Topics/backlog.md`,找待做(非 `published`)条目,列给用户选一个。
- **当场给**:用户提供主题/链接/文本,整理成等价结构(选题/摘要/来源)。

**第 2 步 事实查证**(联网查证开启时)

对关键事实逐条查证(详见 `references/fact-checking.md`)。用 agent 可用的联网搜索能力;无能力则跳过,标"⚠️ 需人工核实"。二手转述不许当原话。

**第 3 步 定核心观点 + 结构 + 字数预算**

- 提炼**一句明确主张**(有立场、有信息量即可,**不强求反常识**)
- 拟结构:开头钩子 → 核心观点 → 分点展开(锚点各一段)→ 结尾互动
- **字数预算分配到每段**(如钩子 250 / 观点 100 / 展开四段各 300 / 结尾 250 = 1600)
- 输出**研究简报(四要素清单)**:
  1. 人设摘要(3-5 行)
  2. 核心观点(1 句)
  3. 结构大纲(每段:标题 + 字数预算 + 一句要点,**不写正文句子**)
  4. 关键事实来源表(事实 / 来源链接 / 一手二手)

> **硬闸门**:用户确认研究简报后才进阶段二。事实或观点没对齐,在此改,不进写作。

### 阶段二:写作

**第 4 步 生成正文**

按研究简报(结构 + 字数预算)+ 人设口吻 + 内置爆款规律(见 `references/explosive-patterns.md`),生成 markdown 正文。agent 自选一个标题作默认,同时附 3-6 个候选。

**第 5 步 自检**

逐项过:
- [ ] 人设口吻(按目的仓库规范)
- [ ] 字数在区间内(`uv run python scripts/wordcount.py <article> --min 1500 --max 2000` 核验)
- [ ] 关键事实均有来源标注
- [ ] 结构完整(钩子/观点/展开/互动)
- [ ] 无禁用词(agent 扫目的仓库禁用词段落;wordcount.py 不做此项)
- [ ] 标题候选 3-6 个

不通过回去改,通过才进第 6 步。

**第 6 步 定版**

列出标题候选 + agent 默认选择,告诉用户"我选了 X,要换吗"。用户确认 → 存 `OUTPUT_DIR/article.md`(frontmatter:`title` / `topic` / `wordcount` / `status: draft`)。提示"可用 md-to-wechat 推草稿"(**不调用**)。

## 输出格式

```markdown
---
title: <agent 选定的标题>
topic: <选题来源>
wordcount: <实际字数>
status: draft
---

# <选定标题>

<正文,markdown,短句+加粗+emoji 分段>

**核心观点**:<一句话>

---

### 参考来源
1. [<文章标题>](链接) — 一手/二手,作者/公众号,日期

### 候选标题(未选用)
- <标题2>
```

## 字数脚本

```bash
uv run python scripts/wordcount.py <article.md> --min 1500 --max 2000
```
退出码 0 = 通过(字数在区间 + 结构标记齐全),1 = 越界或缺标记。定版前必跑。

## 不做(YAGNI)

- 不调用 md-to-wechat(只提示)/ text-to-card(二选一)
- 不依赖红狐等付费 API
- 不写死飞栗人设(通用版)
- 不做禁用词机器检测(归 agent 自检)

## 经验教训(踩坑记录)

- **先控字数再写,不写长再砍** —— 字数预算在第 3 步研究简报就分到每段,第 4 步按预算写;不要"先全写完再删",删比写更容易出错(残留旧行)。
- **核心观点不必反常识** —— AI 领域新事物本身够新,如实陈述其意义就是好长文;硬凑反常识反而失真。
- **二手素材不当原话** —— 知乎/公众号整理的数字和战绩不许当原话引用,查证一手来源。
- **选题从 backlog 不从 inbox** —— inbox 是 harvest-topics 的输入(原始素材),backlog 才是加工过的选题。
- **字数脚本不做禁用词** —— 禁用词在目的仓库是自然语言段落,脚本解析不了;归 agent 自检。
